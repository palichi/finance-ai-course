# 📁 Day 2 · 07 · Fine Tunning· 학습 모델 비교 — RAG에 "학습"을 더하면?

지금까지 만든 RAG는 **학습이 전혀 없는 방식**입니다.
모델(GPT)은 그대로 두고, 검색된 데이터만 매번 프롬프트에 끼워 넣는 구조였죠.

여기서는 "모델 자체를 학습시키는" 3가지 방법을 비교하고,
어떤 상황에 어떤 학습 방식을 쓰는지 정리합니다.

---

#### 🤔 왜 RAG 자료에 LoRA가 없었나?

| | RAG (지금까지 만든 것) | Fine-tuning (LoRA 등) |
|---|---|---|
| 학습 대상 | 없음 (모델 그대로 사용) | 모델 가중치 일부 업데이트 |
| 데이터 갱신 | DB만 새로 넣으면 즉시 반영 | 재학습 필요 |
| 비용/난이도 | API 호출 + 임베딩만 | GPU, 학습 데이터셋, 학습 시간 필요 |

주가처럼 매일 바뀌는 데이터는 "가중치에 박아넣는" fine-tuning보다
"매번 검색해서 갖다 쓰는" RAG가 구조적으로 더 적합합니다.
그래서 이 폴더(`05_rag`)는 학습 없이 RAG만으로 구성되어 있었습니다.

그래도 "학습"을 더해서 RAG를 더 정교하게 만들 수 있는 3가지 방법이 있습니다.

---

#### 📊 3가지 학습 방식 비교

| | 임베딩 Fine-tuning | RAFT | QLoRA |
|---|---|---|---|
| **무엇을 학습시키나** | 검색 단계의 벡터 변환 모델 | 생성 단계의 LLM, "검색결과 중 진짜 근거 골라내기" | 생성 단계의 LLM 전체, 자원 절약형 |
| **목적** | 검색 정확도(Top-K 적중률) 향상 | 답변 품질 + 무관한 문서 무시하는 능력 향상 | LoRA와 동일 목적, GPU 메모리만 적게 |
| **데이터 형태** | (질문, 정답문서) 쌍 수백~수천 개 | (질문, 검색결과 묶음[정답+오답 혼합], 정답) 수천 개 | (질문, 정답) 쌍, 보통 수천~수만 개 |
| **GPU 요구량** | 낮음 (8GB대 가능) | 중~높음 (16GB 이상 권장) | 낮음 (8~12GB, 4bit 양자화) |
| **이 실습과의 연결고리** | STEP 8 미션 1 "k값 슬라이더"와 직결 | STEP 7 "RAG vs 일반 LLM 비교"의 다음 단계 | 4bit 양자화 + LoRA 어댑터 합칠 때 dtype 이슈 주의 |
| **체감 난이도** | ★★☆☆☆ | ★★★★☆ | ★★★☆☆ |

---

#### 🔄 학습 순서 (권장)

```
1단계: RAG만 (STEP 1~8, 이미 완료)
   → 검색이 정확하면 학습 없이도 LLM이 잘 답한다는 것을 체감

2단계: 임베딩 Fine-tuning
   → 검색 자체를 더 정확하게 만들면 어떻게 달라지는지 체감
   → 가장 적은 비용으로 가장 큰 효과 (실무 1순위)

3단계: QLoRA로 생성모델 학습
   → 검색은 그대로 두고, 답변 스타일/말투를 가중치로 학습
   → 예: "20년차 애널리스트 말투"를 프롬프트가 아니라 학습으로 구현

4단계: RAFT
   → 2단계 + 3단계를 합친 것
   → "검색 결과 중 진짜와 가짜를 구분하며 답하는" 가장 어려운 학습
   → 실무에서 RAG 정확도를 한계까지 끌어올릴 때 사용
```

각 단계는 이전 단계의 한계를 보완하는 구조이므로, 순서대로 진행하는 것을 권장합니다.

#### 🖥 [Claude Code 실습] 임베딩 Fine-tuning 맛보기

가장 난이도가 낮고 효과가 큰 **임베딩 Fine-tuning**부터 시작합니다.

**새 터미널 탭**에서 Claude Code 실행 후, 아래 프롬프트를 그대로 복사해서 붙여넣기 하세요:

```
embedding_finetune.py 파일을 만들어줘.

sentence-transformers 라이브러리를 사용해서
한국 주식 도메인에 특화된 임베딩 모델을 fine-tuning하는 파일이야.

기능 1 · 베이스 모델 로딩
  모델명: BAAI/bge-m3
  (없으면 huggingface에서 자동 다운로드)

기능 2 · 학습 데이터 구성
  ../04_chromadb/chroma_db 에서 문서를 가져와서
  아래 형태로 (질문, 정답문서) 쌍 20개를 예시로 직접 생성:
  - "삼성전자 12월 주가는?" ↔ "2024년 12월 31일 삼성전자(005930) 종가 78,500원..."
  - "거래량 가장 많았던 날" ↔ 해당 거래량 데이터 문서
  등 도메인에 맞는 예시로 구성

기능 3 · Contrastive Learning 학습
  sentence-transformers의 MultipleNegativesRankingLoss 사용
  epoch=3, batch_size=8

기능 4 · 학습 전후 비교
  같은 질문 5개에 대해
  - 베이스 모델(bge-m3) 검색 결과 Top-3
  - fine-tuning된 모델 검색 결과 Top-3
  를 나란히 출력해서 유사도 점수 차이를 비교

기능 5 · 학습된 모델 저장
  경로: ./finetuned_embedding/

기타 조건:
  GPU 없으면 CPU로 자동 전환, 소요 시간 경고 출력
  파일명: embedding_finetune.py
```

#### ❓ 자주 묻는 질문

**Q. GPU가 없는데 가능한가요?**
```
임베딩 fine-tuning은 CPU로도 가능합니다 (단, 시간이 더 걸림).
RAFT나 QLoRA는 GPU 권장 — VS Code Remote-SSH로 GPU 서버 연결 후 진행하세요.
```

**Q. 학습 데이터는 몇 개나 필요한가요?**
```
임베딩 fine-tuning: 최소 수백 개 쌍부터 효과 체감 가능
RAFT/QLoRA: 수천 개 이상 권장 (적으면 과적합 위험)
```

**Q. dtype mismatch 오류가 나요**
```
QLoRA(4bit 양자화) 사용 시 자주 발생하는 이슈입니다.
모델 로딩 시 torch_dtype을 통일하세요:
"embedding_finetune.py 에서 torch_dtype=torch.float16 으로
모든 텐서 타입을 통일해줘" 라고 Claude Code에 요청하세요.
```
